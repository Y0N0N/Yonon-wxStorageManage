# 🛠️ Haili 开发技能与工作方式

> 本文档定义 AI 助理在本项目中的**能力边界、工作方式、响应规范**，确保每次交互都聚焦微信小程序开发。

---

## 一、核心能力

### 1.1 代码生成
- ✅ **微信小程序全栈开发**：WXML / WXSS / TypeScript / JSON
- ✅ **数据流设计**：页面数据绑定、组件通信、状态管理
- ✅ **样式与 UI**：Sass 样式、Flex/Grid 布局、动画、Skyline 渲染特性
- ✅ **API 集成**：`wx.request`、云开发 SDK、第三方 API 接入
- ✅ **数据库设计**：数据模型、索引、查询优化
- ❌ **不能做的**：原生 App 开发、非小程序前端框架（React/Vue）

### 1.2 架构设计
- 根据需求设计合理的页面/组件结构
- 设计可扩展的数据模型
- 规划路由与权限体系
- 提出后端方案建议（云开发 / 自建）

### 1.3 代码审查
- 检查是否符合 AGENTS.md 与 _guide/*.md 的规范
- 检查 Skyline 兼容性
- 检查性能（setData 优化、渲染效率）
- 建议最佳实践

---

## 二、工作流程

当接受一个任务时，按照以下顺序执行：

```
Step 1: 读取约束文件
        ├── .agents/AGENTS.md（项目总则）
        └── Haili/_guide/goal.md（功能目标）
        └── Haili/_guide/skills.md（工作方式）

Step 2: 理解需求
        └── 与 goal.md 中的功能范围对照
        └── 明确页面/组件/数据层面影响

Step 3: 规划方案
        └── 输出文件清单与改动说明
        └── 涉及数据模型变更 → 先确认

Step 4: 实现代码
        └── 遵守四件套规范（.wxml / .wxss / .ts / .json）
        └── 每次 setData 精简，合并多次调用
        └── 所有新增页面注册到 app.json

Step 5: 验证
        └── 类型检查（TypeScript）
        └── 逻辑完整性
        └── 对照 goal.md 确认功能覆盖
```

---

## 三、代码输出规范

### 3.1 输出格式
每次代码输出遵循以下格式：
1. **文件路径** — 标明文件位置
2. **改动说明** — 改了什么、为什么
3. **代码内容** — 完整可用的代码
4. **注意事项** — 边界情况、后续优化方向

### 3.2 注释规范
- WXML：组件用途标注
- TS：函数功能、参数说明、复杂逻辑注释
- WXSS：关键样式分组说明
- JSON：配置项简短说明

### 3.3 错误处理
所有 API 调用必须有 `fail` 回调或 `catch`：
```typescript
wx.request({
  // ...
  success: (res) => { /* 正常处理 */ },
  fail: (err) => {
    wx.showToast({ title: '请求失败', icon: 'none' });
    console.error('[API Error]', err);
  }
});
```

## 四、响应风格

### 4.1 主动检查清单
每次输出代码前，自动检查：
- [ ] 文件位置是否符合目录约定？
- [ ] 是否使用了 TypeScript 而非 JS？
- [ ] 是否使用了 .wxss 而非 .css？
- [ ] 新增页面是否注册到 app.json？
- [ ] 是否使用了 `wx.xxx` API 而非 DOM API？
- [ ] `setData` 是否只传递了必要字段？
- [ ] 是否有错误处理？
- [ ] Skyline 兼容性是否验证？

### 4.2 提问原则
需要选择时，AI 会：
1. 给出**推荐方案**并说明理由
2. 列出备选方案及取舍
3. 不追问无意义的问题

### 4.3 解释方式
- 复杂逻辑：拆解步骤 + 代码示例
- 报错原因：定位文件 + 行号 + 修复方案
- 架构建议：图示 + 对比分析

## 五、项目知识地图

AI 维护以下项目状态信息：

| 信息 | 说明 |
|------|------|
| 当前阶段 | 项目处于哪个开发阶段 |
| 已完成的页面 | 页面列表与功能状态 |
| 待实现的功能 | 未完成的 goal.md 功能点 |
| 最新修改 | 最近的代码变动 |
| 已知问题 | 待修复的 bug 或优化项 |

---

## 六、约束优先级

当约束文件之间出现冲突时，优先级从高到低：

```
1. .agents/AGENTS.md（项目总则，最高）
2. Haili/_guide/goal.md（产品目标）
3. Haili/_guide/skills.md（工作方式）
4. Haili/_guide/*.md（其他补充文档）
```

---

> 📌 此文件可与 goal.md 同步更新，以适应项目不同阶段的需求变化。
